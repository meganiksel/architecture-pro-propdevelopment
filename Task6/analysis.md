# Отчёт по результатам анализа Kubernetes Audit Log

## Контекст

Анализ проведён на основе событий, сгенерированных скриптом `simulate-incident.sh` в namespace `secure-ops`.
Скрипт симулировал 5 типов потенциально вредоносных действий в кластере Kubernetes.

---

## Подозрительные события

### 1. Доступ к секретам

- **Кто:** `system:serviceaccount:secure-ops:monitoring`
- **Где:** namespace `kube-system`, ресурс `secrets` (default-token)
- **Действие:** `get` секрета через `kubectl get secret ... --as=system:serviceaccount:secure-ops:monitoring`
- **Почему подозрительно:**
  - ServiceAccount `monitoring` из namespace `secure-ops` не должен иметь доступа к секретам в `kube-system`.
  - Попытка получить `default-token` из `kube-system` — классический вектор атаки для получения привилегированного токена.
  - Доступ к токенам `kube-system` может дать злоумышленнику права cluster-admin.
  - Нарушение принципа минимальных привилегий: ServiceAccount мониторинга не должен читать секреты других namespace.

---

### 2. Привилегированные поды

- **Кто:** Пользователь, выполнивший скрипт (текущий контекст kubectl)
- **Что:** Создан под `privileged-pod` с `securityContext.privileged: true`
- **Комментарий:**
  - Привилегированный под имеет полный доступ к хосту: файловой системе, сети, устройствам.
  - Это эквивалентно запуску процесса с правами root на узле кластера.
  - Злоумышленник может выйти за пределы контейнера (container escape) и скомпрометировать весь узел.
  - Создание привилегированных подов должно быть заблокировано политиками PodSecurity/OPA Gatekeeper.
  - **Оценка:** Критическая угроза — компрометация узла кластера.

---

### 3. Использование kubectl exec в чужом поде

- **Кто:** Пользователь, выполнивший скрипт (текущий контекст kubectl)
- **Что делал:** Выполнил `kubectl exec` в поде `coredns` в namespace `kube-system` — запустил `cat /etc/resolv.conf`
- **Почему подозрительно:**
  - `coredns` — критический системный компонент кластера. Несанкционированный exec в него недопустим.
  - Через exec в системный под злоумышленник может изучить внутреннюю сеть кластера, DNS-конфигурацию.
  - Возможна модификация конфигурации DNS для перехвата трафика (DNS poisoning).
  - Нарушение принципа разделения привилегий: пользователи не должны иметь exec-доступ к системным подам.

---

### 4. Создание RoleBinding с правами cluster-admin

- **Кто:** Пользователь, выполнивший скрипт (текущий контекст kubectl)
- **Что:** Создан `RoleBinding` с именем `escalate-binding`, привязывающий ServiceAccount `monitoring` к `ClusterRole: cluster-admin`
- **К чему привело:**
  - ServiceAccount `monitoring` получил права **cluster-admin** — полный контроль над кластером.
  - Это классическая атака **privilege escalation** (повышение привилегий).
  - Злоумышленник, скомпрометировавший ServiceAccount `monitoring`, теперь имеет неограниченный доступ ко всем ресурсам кластера.
  - Создание RoleBinding без согласования — нарушение процесса управления изменениями.
  - **Оценка:** Критическая угроза — полная компрометация кластера.

---

### 5. Удаление audit-policy.yaml

- **Кто:** Пользователь с флагом `--as=admin` (попытка имперсонации)
- **Что:** Попытка удалить файл `/etc/kubernetes/audit-policy.yaml` через `kubectl delete`
- **Возможные последствия:**
  - Отключение аудита кластера — злоумышленник пытается «замести следы».
  - Без audit-policy все последующие действия не будут логироваться.
  - Попытка имперсонации `--as=admin` указывает на осведомлённость атакующего о механизмах авторизации.
  - Даже неудачная попытка удаления — индикатор компрометации: легитимный пользователь не должен пытаться удалить политику аудита.

## Ошибки политики RBAC

| № | Ошибка | Описание | Рекомендация |
|---|--------|---------|-------------|
| 1 | Избыточные права ServiceAccount | `monitoring` SA смог запросить секреты `kube-system` | Применить принцип минимальных привилегий: SA мониторинга должен иметь доступ только к метрикам |
| 2 | Отсутствие запрета на создание привилегированных подов | Привилегированный под создан без ограничений | Включить PodSecurity Admission с уровнем `restricted` или `baseline` |
| 3 | Разрешён exec в системные поды | Пользователь выполнил exec в `coredns` | Запретить `pods/exec` для всех кроме cluster-admin; использовать NetworkPolicy |
| 4 | Возможность создания RoleBinding без ограничений | Создан RoleBinding с cluster-admin без согласования | Запретить создание RoleBinding/ClusterRoleBinding без явного согласования через GitOps/OPA |
| 5 | Имперсонация пользователей | Попытка `--as=admin` не заблокирована на уровне политик | Ограничить право `impersonate` только для системных компонентов |
| 6 | Отсутствие мониторинга привилегированных действий | Все инциденты произошли без алертов | Настроить SIEM/alerting на критические события audit.log |

---

## Что можно считать компрометацией кластера

1. **Создание RoleBinding `escalate-binding`** — ServiceAccount `monitoring` получил права cluster-admin. Кластер **скомпрометирован**: злоумышленник имеет полный контроль.
2. **Создание привилегированного пода** — возможен container escape и компрометация узла.
3. **Попытка удаления audit-policy** — попытка скрыть следы атаки.

## Вывод

В ходе симуляции инцидента выявлены **5 критических событий безопасности**, из которых 2 привели к фактической компрометации кластера (privilege escalation через RoleBinding и создание привилегированного пода). Атака следовала классической цепочке: разведка (доступ к секретам) → закрепление (привилегированный под) → повышение привилегий (RoleBinding с cluster-admin) → сокрытие следов (удаление audit-policy).

**Немедленные меры:**
1. Удалить `escalate-binding` RoleBinding.
2. Удалить `privileged-pod`.
3. Отозвать все токены ServiceAccount `monitoring`.
4. Включить PodSecurity Admission с уровнем `restricted`.
5. Настроить OPA Gatekeeper для запрета создания RoleBinding без согласования.
6. Настроить алертинг на критические события в audit.log.
