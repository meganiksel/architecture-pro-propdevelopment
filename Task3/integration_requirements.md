# Требования к внешним интеграциям: Умный дом (PropDevelopment)

## Контекст

PropDevelopment внедряет два новых сервиса «Умного дома» через партнёрскую платформу:
- **Интеллектуальный домофон** — видеодомофон с биометрическим распознаванием лиц
- **Интеллектуальный шлагбаум** — управление шлагбаумом с распознаванием номеров автомобилей

Новые сервисы доступны через существующее мобильное приложение собственников.

## 1. Требования к безопасности

### 1.1 Защита персональных данных и биометрии

| ID | Требование | Приоритет |
|----|-----------|-----------|
| SEC-01 | Биометрические данные (изображения лиц, шаблоны) классифицируются как **специальная категория ПДн** согласно ст. 11 152-ФЗ. Обработка допускается только с явного письменного согласия субъекта. | Критический |
| SEC-02 | Данные о номерах автомобилей классифицируются как **персональные данные** (позволяют идентифицировать владельца). Обработка — по 152-ФЗ. | Критический |
| SEC-03 | Биометрические шаблоны **не передаются** партнёру в открытом виде. Передача только зашифрованных хешей или токенов доступа. | Критический |
| SEC-04 | Хранение биометрических данных осуществляется **только на серверах PropDevelopment** (или в сертифицированном облаке на территории РФ). Партнёр не хранит биометрию. | Критический |
| SEC-05 | Все данные умного дома (конфигурации, списки доступа, журналы событий) хранятся в отдельной БД `smart-home-db` с шифрованием at-rest (AES-256). | Высокий |
| SEC-06 | Реализована изоляция данных по жилым комплексам: партнёр одного ЖК **не может** получить данные другого ЖК. Изоляция обеспечивается на уровне `smart-home-gateway`. | Критический |
| SEC-07 | Реализована процедура удаления биометрических данных при отзыве согласия или выезде собственника. Срок исполнения — не более 72 часов. | Высокий |
| SEC-08 | Контракты API партнёра проверены на минимизацию передаваемых данных (data minimization). Передаются только данные, необходимые для выполнения конкретной функции. | Высокий |

### 1.2 Защита от несанкционированного доступа

| ID | Требование | Приоритет |
|----|-----------|-----------|
| SEC-09 | Все запросы к API партнёра проходят через `smart-home-gateway`. Прямые запросы от мобильного приложения к партнёру **запрещены**. | Критический |
| SEC-10 | Реализована защита от атак повторного воспроизведения (replay attacks) при передаче команд управления устройствами. | Высокий |
| SEC-11 | Реализован механизм подтверждения действий для критических операций (открытие двери, управление шлагбаумом) — push-уведомление или PIN. | Высокий |
| SEC-12 | Настроен rate limiting: не более 10 запросов в минуту на одного пользователя к API умного дома. | Средний |
| SEC-13 | Реализован аудит всех действий: кто, когда и какую команду отправил устройству. Журнал хранится не менее 1 года. | Высокий |
| SEC-14 | Реализована защита от IDOR (Insecure Direct Object Reference): пользователь может управлять только устройствами своего ЖК и квартиры. | Критический |

### 1.3 Безопасность инфраструктуры

| ID | Требование | Приоритет |
|----|-----------|-----------|
| SEC-15 | `smart-home-gateway` развёртывается в отдельном namespace Kubernetes с сетевыми политиками, ограничивающими входящий и исходящий трафик. | Высокий |
| SEC-16 | Секреты (API-ключи партнёра, сертификаты) хранятся в Kubernetes Secrets или Vault. Не допускается хранение в переменных окружения или конфигурационных файлах. | Критический |
| SEC-17 | Образ контейнера `smart-home-gateway` проходит сканирование на уязвимости (CVE) перед деплоем в production. | Высокий |
| SEC-18 | Реализован circuit breaker: при недоступности платформы партнёра система деградирует gracefully (ручное управление остаётся доступным). | Средний |

## 2. Протоколы аутентификации и авторизации

### 2.1 Аутентификация пользователей (собственников)

| ID | Требование | Детали |
|----|-----------|--------|
| AUTH-01 | Аутентификация собственников в мобильном приложении — через существующий `auth-service-1` (Keycloak). | OAuth 2.0 + OpenID Connect |
| AUTH-02 | Для доступа к функциям умного дома требуется **дополнительное подтверждение** (step-up authentication): биометрия устройства или PIN. | FIDO2 / PIN |
| AUTH-03 | JWT-токены для доступа к `smart-home-gateway` имеют срок жизни не более **15 минут**. Refresh token — не более 24 часов. | OAuth 2.0 |
| AUTH-04 | В JWT-токене содержится claim с идентификатором ЖК (`complex_id`) и квартиры (`apartment_id`) для авторизации на уровне ресурсов. | JWT Claims |

### 2.2 Аутентификация сервис-к-сервису (PropDevelopment ↔ Партнёр)

| ID | Требование | Детали |
|----|-----------|--------|
| AUTH-05 | Взаимодействие `smart-home-gateway` с платформой партнёра использует **mTLS** (mutual TLS) для взаимной аутентификации сторон. | TLS 1.3, X.509 |
| AUTH-06 | Дополнительно к mTLS используется **OAuth 2.0 Client Credentials Flow** для авторизации запросов от `smart-home-gateway` к партнёру. | OAuth 2.0 |
| AUTH-07 | API-ключи и сертификаты ротируются не реже одного раза в **90 дней**. | Автоматическая ротация |
| AUTH-08 | Партнёр предоставляет отдельные credentials для каждого жилого комплекса. Компрометация одного ЖК не затрагивает другие. | Изоляция по ЖК |

### 2.3 Авторизация (управление доступом)

| ID | Требование | Детали |
|----|-----------|--------|
| AUTH-09 | Реализована модель RBAC для управления устройствами умного дома: роли `owner` (полный доступ к своей квартире), `building_manager` (управление общедомовыми устройствами), `admin` (конфигурация). | RBAC |
| AUTH-10 | Авторизация на уровне ресурсов (ABAC): доступ к устройству разрешён только если `user.complex_id == device.complex_id AND user.apartment_id == device.apartment_id`. | ABAC |
| AUTH-11 | Список разрешённых лиц (биометрия) и автомобилей (номера) управляется только собственником квартиры или уполномоченным менеджером ЖК. | RBAC + ABAC |

## 3. Организация взаимодействия между системами

### 3.1 Архитектурные решения

```
Мобильное приложение
        │
        │ HTTPS + JWT (OAuth 2.0)
        ▼
tenant-core-app
        │
        │ REST + JWT (внутренний)
        ▼
smart-home-gateway  ◄──── smart-home-db (PostgreSQL)
        │
        │ HTTPS + mTLS + OAuth 2.0 Client Credentials
        ▼
Платформа партнёра (Умный дом)
        │
        ├──► Интеллектуальный домофон
        └──► Интеллектуальный шлагбаум
```

### 3.2 Требования к взаимодействию

| ID | Требование | Детали |
|----|-----------|--------|
| INT-01 | Все коммуникации используют **TLS 1.3** (минимум TLS 1.2). Устаревшие протоколы (SSL, TLS 1.0/1.1) запрещены. | TLS |
| INT-02 | `smart-home-gateway` реализует паттерн **API Gateway**: маршрутизация, аутентификация, авторизация, rate limiting, логирование — централизованно. | API Gateway |
| INT-03 | Взаимодействие с партнёром — **синхронное** (REST/HTTPS) для команд управления, **асинхронное** (Webhook/событийная модель) для уведомлений о событиях устройств. | REST + Webhook |
| INT-04 | Партнёр обязан предоставить **SLA**: доступность не менее 99.5%, время ответа API не более 2 секунд. | SLA |
| INT-05 | Реализован **идемпотентный** механизм отправки команд: повторная отправка одной команды не приводит к двойному выполнению. | Idempotency Key |
| INT-06 | Все запросы к партнёру логируются в `smart-home-db` с указанием: timestamp, user_id, complex_id, device_id, action, response_code. | Аудит |
| INT-07 | Реализован **health check** для мониторинга доступности платформы партнёра. При недоступности — алерт в систему мониторинга PropDevelopment. | Мониторинг |
| INT-08 | Контракт API партнёра фиксируется в виде OpenAPI 3.0 спецификации. Изменения контракта согласовываются с командой безопасности PropDevelopment. | API Contract |
| INT-09 | Реализована **версионность API**: при изменении контракта партнёра поддерживается обратная совместимость не менее 6 месяцев. | Версионирование |
| INT-10 | Данные о событиях устройств (открытие двери, проезд автомобиля) передаются в DWH PropDevelopment через CDC для аналитики и аудита. | CDC → DWH |

### 3.3 Требования к договору с партнёром

| ID | Требование |
|----|-----------|
| LEGAL-01 | Договор с партнёром включает **DPA (Data Processing Agreement)** с требованиями по защите ПДн согласно 152-ФЗ. |
| LEGAL-02 | Партнёр обязуется **не передавать** данные собственников третьим лицам без согласия PropDevelopment. |
| LEGAL-03 | Партнёр обязуется уведомлять PropDevelopment об инцидентах безопасности в течение **24 часов**. |
| LEGAL-04 | PropDevelopment имеет право проводить **аудит безопасности** платформы партнёра не реже одного раза в год. |
| LEGAL-05 | При расторжении договора партнёр обязуется **удалить все данные** PropDevelopment в течение 30 дней с подтверждением. |

## 4. Новые компоненты в ландшафте систем

### 4.1 `smart-home-gateway` (новый контейнер)

- **Тип:** Container (Kotlin, SpringBoot)
- **Назначение:** API-шлюз для изоляции взаимодействия с партнёром
- **Функции:**
  - Аутентификация и авторизация запросов от `tenant-core-app`
  - Трансляция внутренних запросов в формат API партнёра
  - Изоляция данных по ЖК (multi-tenancy)
  - Аудит всех операций
  - Rate limiting и circuit breaker

### 4.2 `smart-home-db` (новая БД)

- **Тип:** Container (PostgreSQL)
- **Назначение:** Хранение данных умного дома
- **Данные:**
  - Конфигурация устройств по ЖК
  - Списки доступа (биометрические токены, номера авто)
  - Журнал событий устройств
  - Журнал команд пользователей
- **Безопасность:** Шифрование at-rest, доступ только из `smart-home-gateway`
